## 3. Operations

Instead of relying on a static `.env` file, use the Infisical CLI to inject secrets directly into the Docker Compose process. This ensures that `docker-compose.yaml` variables (like `${EMCP_GITHUB_SECRET}`) are populated with fresh values from the vault.

### Start the Stack

Run the following command to inject `prod` secrets from the `/emcp` path:

```bash
infisical run --env=prod --path=/emcp -- docker compose up -d
```

### Rebuild and Start

If you have changed Dockerfiles or images:

```bash
infisical run --env=prod --path=/emcp -- docker compose up -d --build
```

### Stop the Stack

Standard docker compose commands work, but it's good practice to keep the context consistent:

```bash
docker compose down
```

> **Note on Defaults**: Some variables in `docker-compose.yaml` have default values (e.g., `${POSTGRES_PASSWORD:-emcp}`). Providing them via Infisical will override these defaults, which is recommended for production security.

### Adding a New Secret

To add a secret (e.g., for a new MCP tool) via the CLI:

```bash
# Syntax: infisical secrets set NAME=VALUE --env=prod --path=/emcp
infisical secrets set NEW_TOOL_KEY=sk-12345 --env=prod --path=/emcp
```

## 5. Troubleshooting

* **"Variable is not set"**: If Docker complains about a missing variable, verify that:
    1. The secret exists in the `prod` environment in Infisical.
    2. The secret is located specifically at the `/emcp` path (not root).
    3. You are running the `infisical run ...` command, not just `docker compose up`.
* **Permission Denied**: Ensure the user running the command has read access to the Infisical workspace.
* **Verification**: To verify what variables are being passed (BE CAREFUL - SHOWS SECRETS):

    ```bash
    infisical run --env=prod --path=/emcp -- env | grep EMCP_
    ```
